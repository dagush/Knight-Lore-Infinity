Coords:
My student detected that the exact expressions to go from low-resolution integer coordinates, where objects are initialised in memory, are translated to high-resolution coordinates are:
    X_h = X_L * 16 + X_l * 8 + 72
    Y_h = Y_L * 16 + Y_l * 8 + 72
    Z_h = Z_L * 12 + Z_l * 4 + ScreenZ
Where X_L, Y_L and Z_L are the low resolution coordinates, the _l is information related to the sprites and allow to work with self-moving objects, and ScreenZ is always 128.

Main Character:
    The head location information is located at 5C20h, 5C2Ah and 5C2Bh for X, Y and Z.
    The body location information is at         5C09h, 5C0Ah and 5C0Bh.
    Sprites, as stored in 0x5C08 (body) and 0x5C28 (head):
        0x10 to 0x15: human body west (6 sprites)
        0x18 to 0x1D: human body east
        0x20 to 0x27: human head west (8 sprites)
        0x28 to 0x2F: human head east
        0x30 to 0x35: wolf body west (6 sprites)
        0x38 to 0x3F: wolf body east
        0x40 to 0x47: wolf head west (8 sprites)
        0x48 to 0x4F: wolf head east
        0x5C to 0x5F: transformation (4 sprites)
    Orientation Summary:
        Positions 5C41h and 5C45h indicate the head and body sprite codes.
        Positions 5C0Fh and 5C2Fh contain the mirror flags for orientation.
        0x5C41 == 0x5C45: 0x18 human west/north, 0x19 human east/south,
                          0x1D wolf west/north,  0x1E wolf east/south.
        0x5C0F >= 0x4C -> south/north, else: west/east
        0x5C2F == 0x7E wolf north
    Pseudocode for character orientation:
        sprite_body = contents of 0x5c41
        if sprite_body is 0x18 or 0x19: body is human, else: body is wolf
        if contents of 0x5c0f >= 0x4c:
            direction is either North or South
            if sprite_body is 0x18 or 0x1d:
                direction is North,
            else: direction is South
        else:
            direction is either west or east
            if sprite_body is 0x18 or 0x1d:
                direction is West,
            else: direction is east
    Renderer compass/view convention:
        View names describe the side where the camera is placed, not the direction it looks.
        North view means the camera is north of the room/character and looks inward, toward the center.
        Therefore, if the character is facing north and the camera side is north, the rendered billboard should show the front.
        If the character is facing north and the camera side is south, it should show the back.
        The same rule is used for east, west, and south.
        The camera side is computed from the room center to the camera, not from the moving character to the camera.
        This keeps the chosen billboard side stable while the character walks.
        Diagonal full-3D character billboard policy:
            Upper south east (game): east-facing billboard, live game sprite ids.
            Upper north west: west-facing billboard, opposite live sprite side, final texture mirror in X.
            Upper south west: south-facing billboard, relative sprite side.
            Upper north east: north-facing billboard, opposite relative sprite side, final texture mirror in X.
        Final X mirroring also includes the live character mirror flags:
            body uses 0x5C0F;
            head uses 0x5C2F;
            transformation sprites use either flag.
        Policy mirror and live mirror are combined as toggles, so two mirrors cancel.

Room/location shape and background presets:
     In the static location entry, byte +2 combines the room size selector with the
     low 3-bit attribute/colour value. The background ids follow this byte.
     If foreground block data exists, the background list is followed by 0xFF
     and then the block/count groups.
     If there is no foreground block data, do not write a 0xFF separator. Let
     the entry end after the background ids. Original room 0x77 follows this
     background-only pattern. Adding 0xFF with no block bytes after it can make
     retrieve_screen enter the foreground decoder and read the next room id as
     block data, which can crash or reset the game.
     The visual wall shape is not controlled by a separate hidden attribute: the
     size selector and the chosen wall/background preset ids must agree.
     This explains the Stage 3 test where 0x5BAB..0x5BAE correctly reported a
     narrow room, but the room looked square because it still used square/tree
     background ids.
     Confirmed size selectors:
        selector 0: 64, 64, 128
        selector 1: 32, 64, 128
        selector 2: 64, 32, 128
     Original static-location patterns:
        selector 0: square rooms. Common wall/background ids include 0x0C and
                    tree room combinations using 0x0F, 0x10, and 0x11.
        selector 1: rectangular 32x64 rooms. These use wall id 0x0E and only
                    north/south exits on the narrow ends.
        selector 2: rectangular 64x32 rooms. These use wall id 0x0D and only
                    east/west exits on the narrow ends.
     Practical rule for injected rooms:
        For selector 1, use north/south exits and wall id 0x0E.
        For selector 2, use east/west exits and wall id 0x0D.
        Avoid doors on the long sides of rectangular rooms.
     Wooden/tree caveat:
        The original data only uses the tree/wood backgrounds 0x0F, 0x10, and
        0x11 with square rooms. There do not appear to be stock rectangular
        tree/wood wall definitions, so rectangular wooden rooms would require
        custom background definitions later.
        Current experimental Stage 7 rectangular tree rooms install custom
        background definitions in the unused tail of the erased location table
        and repoint background ids 0x12 and 0x13 to them:
           0x12: custom tree wall size 2, for selector 2 rooms.
           0x13: custom tree wall size 3, for selector 1 rooms.
        They also use tree arch ids 0x04..0x07 for legal narrow-side exits.
        Their custom wall records keep only source records that already use
        Z/base byte 0x80. Raised stone-wall layers are discarded rather than
        moved down, because lowering them can cover otherwise valid arches.
        The current long-wall fill strategy copies the complete original
        square-tree wall pattern and translates only the room-size plane:
           selector 1 / custom 0x13:
              full west wall pattern, x moved from 0x3F to 0x5F.
              rows use y = 0x49, 0x58, 0x68, 0x78, 0x88, 0x98, 0xA8, 0xB8.
           selector 2 / custom 0x12:
              full north wall pattern, y moved from 0xC0 to 0xA0.
              rows use x = 0x48, 0x58, 0x68, 0x78, 0x88, 0x98, 0xA8, 0xB8.
        The runtime removes the matching stone-derived ground wall records
        before appending these translated tree records, to avoid duplicate
        overlapping panels.
        Earlier attempts to use only the two filler records left holes.
        Lowering raised stone layers could cover valid arches, and hand-filled
        new explicit records caused unstable player/emulator behaviour. These
        attempts were reverted.
        This is new Knight Lore-compatible data, not a pattern observed in the
        original room dump. Outside Stage 7, 0x12/0x13 should still be treated
        cautiously because they are the original wizard/cauldron ids.
     Fixed-size entry caveat:
        When testing by rewriting a room entry without changing its total byte
        length, preserve the number of background slots before 0xFF. Cleaned
        null expansion ids such as 0x12 and 0x13 can be used as fillers after
        the wizard/cauldron expansion data has been zeroed.
        Example selector 1 background list: 0x00, 0x02, 0x0E, 0x12, 0x13, 0x12, 0x13, 0xFF
        Example selector 2 background list: 0x01, 0x03, 0x0D, 0x12, 0x13, 0x12, 0x13, 0xFF

Working memory: between 5BA0h and 6107h

Live graphic object table:
     Starts at 0x5C08, with up to 40 slots of 0x20 bytes each.
     Slot layout, from the disassembly:
        +0 graphic/sprite id
        +1, +2, +3: X, Y, Z
        +4, +5, +6: width/radius X, depth/radius Y, height
        +7 flags:
            bit 7 = vertical flip
            bit 6 = horizontal flip
            bit 5 = wipe, transient
            bit 4 = draw, transient
            bit 3 = auto-adjust near arches, player only
            bit 2 = moveable
            bit 1 = ignore in 3D calculations
            bit 0 = near arch, player only
        +8 room
        +9, +10, +11: dX, dY, dZ as signed bytes
        +13 per-object direction/state information
     Confirmed guard/wizard slots:
        0x60C8: guard slot. Coordinates and +7 flags are confirmed correct.
        0x6028 and 0x6048: second guard type, top and lower halves. Coordinates are confirmed correct.
            This guard walks east/west, so +7 bit 6 does not toggle; the apparent lack of hflip change is expected.
            The east/west visual side is represented by the sprite id/dX logic, not by hflip.
        Wizard in the cauldron room: coordinates are confirmed in the dynamic entries paired with background rows 8 and 9.
            The same +7 flag layout must be used for the wizard; if a row comes from compact comparison data,
            read the raw flag byte from offset +7.
     Direction/state note:
        For NSEW guard/wizard movement, +13 low two bits decode as:
            0 = west, 1 = north, 2 = east, 3 = south.
