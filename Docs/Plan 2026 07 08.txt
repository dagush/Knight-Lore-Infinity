Plan 2026 07 08
================

Dispersed Quest Sectors and External Dialogue Plan
--------------------------------------------------

Purpose
-------

This plan captures the next design direction after the Stage 7 sliding
five-room cross proof.

The infinite map is no longer only a technical trick for making Knight Lore
larger. It becomes a new game structure: a dispersed world made of local
8x8 quest sectors. Each sector can contain its own local cauldron and a nearby
matching charm, turning the original global "find all charms for one cauldron"
structure into many smaller local quests spread across the infinite map.

The plan keeps the existing hard boundary:

- do not modify the original Knight Lore source code;
- do not add a new renderer;
- use the original Spectrum/JSSpeccy display for the game;
- put new world logic, persistence, diagnostics, and dialogue in JavaScript;
- interact with the original game only through emulator memory inspection,
  static-table rewriting, working-memory rewriting, and careful runtime pokes.


Current Foundation
------------------

Already working:

- JSSpeccy loads Knight Lore by default from the local Z80 snapshot.
- The project has a local `_test` setup for browser testing.
- Static room/location data can be decoded and displayed.
- Physical room 0x88 can be forced as the practical starting room.
- The static location table can be erased and replaced with controlled room
  payloads.
- A five-room physical cross can be injected:
  - center: 0x88;
  - west: 0x87;
  - east: 0x89;
  - south: 0x78;
  - north: 0x98.
- Stage 7 can keep the player effectively centered on physical 0x88 while
  logical coordinates move over an infinite deterministic map.
- The generator supports:
  - logical coordinates `(x, y)`;
  - 8x8 chunks;
  - deterministic chunk connectivity;
  - reciprocal doors;
  - missing rooms;
  - biome/style regions;
  - square and original-compatible rectangular rooms;
  - experimental wooden rectangular rooms using custom background records.

Important correction:

- The "do not place anything in the spawn/center clearance area" rule is only
  strictly required for the starting room, physical/logical 0x88.
- For every other room, the stronger rule is to keep entrance strips safe so
  that the player is not trapped behind walls or solid blocks after crossing a
  room boundary.


High-Level Game Idea
--------------------

Original Knight Lore:

- uses a dense 16x16 possible room grid;
- contains one cauldron;
- has a set of possible charm/item spawn locations;
- asks the player to bring 14 objects to the cauldron, using seven actual
  charm types requested twice. The nearby 0x67 object code is extra life, not
  a quest charm.

Dispersed Knight Lore Infinity:

- uses the infinite logical map already provided by Stage 7;
- divides the map into deterministic local quest sectors, probably 8x8 rooms;
- each sector may contain:
  - one local cauldron room;
  - one local charm room;
  - one required charm type;
  - local persistent quest state;
- the player travels through the infinite map completing many local charm
  deliveries instead of servicing only the original cauldron.

This turns the infinite-map system into a gameplay layer:

- explore a sector;
- locate its cauldron;
- learn which charm is required;
- find that charm somewhere in the same sector;
- bring it back;
- move on to another sector.

The cauldron and charm belong to the same sector, but the route between them
does not have to remain inside that sector. A valid path may temporarily leave
the sector and come back through neighbouring sectors. This is desirable: it
reduces pressure on each 8x8 sector to be internally complete and makes nearby
sectors feel more naturally integrated.

The charm type itself is global. If the player already carries the charm type
requested by a cauldron, that charm should be usable even if it was collected
in another sector. The local charm anchor is therefore a suggested nearby
source, not an enforced sector-bound item.


Design Vocabulary
-----------------

Logical room:

- A JS-side room payload at coordinate `(x, y)`.
- It compiles into one of the five physical Knight Lore location entries.
- It may contain topology, style, blocks, room labels, and future quest hooks.

Chunk:

- The existing 8x8 procedural generation unit.
- Used for topology and connectivity.

Quest sector:

- A gameplay region.
- Recommended first version: use the same 8x8 coordinate grid as chunks.
- Later versions may group multiple chunks, but sharing the 8x8 chunk boundary
  keeps the first proof simple.

Sector coordinate:

- For room `(x, y)` and sector size 8:
  - `sectorX = floorDiv(x, 8)`;
  - `sectorY = floorDiv(y, 8)`;
  - `localX = mod(x, 8)`;
  - `localY = mod(y, 8)`.

Quest anchor room:

- A room that the quest layer requires to exist.
- The first two anchor types are:
  - cauldron anchor;
  - charm anchor.

Persistent quest state:

- JS-side state keyed by sector coordinate.
- It must survive room sliding, room cache eviction, and eventually save/load.


Stage 8 - Dispersed Quest Sector Model
--------------------------------------

Goal:

- Define deterministic per-sector quest metadata without changing gameplay
  memory yet.

Tasks:

- Add a quest-sector model in JavaScript, preferably separate from the low-level
  room injection code.
- Use deterministic functions of `(sectorX, sectorY, worldSeed)` to pick:
  - whether the sector has a quest;
  - required charm type;
  - cauldron local coordinate;
  - charm local coordinate;
  - optional difficulty/style modifiers.
- Force cauldron and charm anchor rooms to exist in generated topology.
- Store persistent state outside the emulator:
  - `unvisited`;
  - `cauldronSeen`;
  - `charmSeen`;
  - `charmCollected`;
  - `completed`.
- Add diagnostics showing the current room's sector metadata.

Recommended first proof:

- Do not inject actual charms or cauldrons into game memory yet.
- Treat cauldron/charm placement as metadata only.
- Show them in the HTML diagnostics table so the algorithm can be inspected.

Exit criteria:

- JS can ask:
  - "what quest sector contains room `(12, -4)`?"
  - "which charm does it require?"
  - "where is its cauldron?"
  - "where is its charm?"
- The answer is deterministic and stable across page reloads.
- The required cauldron/charm rooms are guaranteed to exist.


Stage 8.1 - Sector Reachability Guarantee
-----------------------------------------

Goal:

- Ensure every generated local quest is fair: the player can reach the cauldron
  and the charm, even if the route temporarily leaves the owning sector.

Tasks:

- Extend or post-process chunk generation so quest anchors are connected to the
  world path network.
- Ensure at least one path exists between:
  - the wider generated map;
  - cauldron room;
  - charm room.
- Prefer an in-sector path when it falls out naturally, but do not require one.
- Allow detours through neighbouring sectors when that produces a valid route
  with less forced carving.
- Keep reciprocal exits by construction.
- Keep rectangular-room legality:
  - selector 1 only for north/south exits;
  - selector 2 only for east/west exits;
  - fall back to square rooms when the exit mask is not rectangular-safe.

Recommended first proof:

- Generate quest anchors after the base chunk core exists.
- If an anchor lands on a missing room, force that room to exist.
- If an anchor is isolated, connect it to the nearest suitable existing path.
  The nearest path may be inside the same sector or across a sector boundary.
- If the anchor lands on an unsuitable room shape, force it to a square room.

Exit criteria:

- A diagnostic reachability check can confirm:
  - cauldron anchor exists;
  - charm anchor exists;
  - both are in the same connected component as the traversable world.
- First implementation note:
  - use a bounded world-graph search around the sector anchors;
  - cache the result per sector;
  - report anchor existence, path length, visited/searched rooms, and search
    margin in diagnostics.
- Honesty note:
  - this is a practical diagnostic over bounded generated regions, supported by
    the chunk-connector invariant, not a formal mathematical proof over the
    infinite plane;
  - manual testing and sampled-sector scans can make reachability failures very
    unlikely, but cannot prove them impossible.


Stage 8.2 - Cauldron And Charm Room Dressing
--------------------------------------------

Goal:

- Make quest rooms visibly identifiable using original Knight Lore room data,
  but still avoid touching the fragile charm/cauldron mechanics at first.

Tasks:

- Mark cauldron and charm rooms in the logical payload.
- Give quest rooms distinct labels and diagnostic metadata.
- Add conservative visual dressing:
  - cauldron room: open, calm, readable, low hazard;
  - charm room: slightly more guarded or puzzle-like;
  - avoid blocking entrance strips.
- Avoid the center/spawn-clearance rule except for the true starting room
  0x88, but still keep the actual player entry side clear.

Possible visual implementations:

- Phase A: diagnostics only.
  - Add `questDressing` metadata to generated room payloads.
  - Label anchor rooms distinctly, for example `quest-cauldron:...` and
    `quest-charm:...`.
  - Show the intended dressing profile in the HTML diagnostics table.
  - Do not alter room bytes, object records, item slots, cauldron state,
    wizard state, inventory state, or charm-order memory.
- Phase B: use safe static room-entry dressing if confirmed harmless.
  - Force generated cauldron/charm anchor rooms to square selector 0 for the
    first visual proof, even if the base topology could have produced a
    rectangular room.
  - Use only the compact location-entry fields already proven by Stage 7:
    header colour/attribute and foreground block runs.
  - Cauldron marker: low-hazard colour plus ordinary block type `0x00` in a
    clear, symmetric four-corner pattern.
  - Charm marker: stronger colour plus spike block type `0x05` in the same
    entry-lane-safe pattern.
  - Keep x=3..4 and y=3..4 central entry lanes clear so any actual player
    entry side remains usable.
  - Still do not use original cauldron/wizard background ids, original item
    slots, original object table records, inventory bytes, or charm-order
    memory.
- Phase C: inject actual original cauldron/charm objects after memory probes.
  Split this into smaller probes rather than treating Phase C as one jump:
  - C1: original cauldron visual background only.
    - Restore the original 0x13 cauldron background expansion after confirming
      the bytes by read-only snapshot probes.
    - Append background id 0x13 only to generated quest cauldron anchor rooms.
    - Force those rooms to selector 0 and a calm colour for the first proof.
    - Suppress the Phase B four-corner marker in cauldron rooms when the
      original visual is active.
    - Keep original item slots, charm-order bytes, inventory bytes, wizard
      state, and cauldron-completion state untouched.
  - C2: bubble template probe.
    - Keep bubbles disabled by default.
    - Preserve `?stage8cbubbles=1` / `?stage8cbubbles=global` as a high-risk
      control that restores the template globally. Manual testing already
      showed this creates bubbles in every room.
    - Add `?stage8cbubbles=cauldron` as a timing probe: restore the template
      only while the current logical center room is a quest cauldron, and zero
      it elsewhere.
    - If repeated `stage8cbubbles` parameters appear in a URL, use the last
      value and report the full request chain in diagnostics.
    - Do not clear live slots yet. If bubbles leak after room exit, add a
      separate live-slot probe for the working area around 0x5C68.
    - Treat all bubble modes as high risk because bubbles are tied to
      wolf-form attack logic.
  - C3: actual charm/item/cauldron mechanic probe.
    - Do this only after the visual and bubble probes show which bytes are
      purely visual and which are stateful.
    - Split C3 into staged probes:
      - C3.0 read-only original mechanic anatomy;
      - C3.1 single charm-room item injection;
      - C3.2 carried/inventory state probe;
      - C3.3 cauldron acceptance probe;
      - C3.4 original cauldron request display/order patch;
      - C3.5 animation/display timing probe, including the fast bubble cycle
        and fast requested-charm flashing.

Risks:

- The original cauldron and wizard records may have special-case code paths.
- Simply duplicating original cauldron data may confuse the original quest
  logic if we do not control related state bytes.
- Charm objects may be coupled to original inventory/order mechanics.

Exit criteria:

- The player can identify a sector's cauldron/charm rooms by diagnostics and,
  later, by safe in-game dressing.
- No room reset, freeze, or invalid retrieve_screen decode occurs.

Stage 8.3 - External Wizard Dialogue
------------------------------------

Goal:

- Add an external dialogue layer that introduces and explains the new dispersed
  quest mechanic without changing the original renderer.

Concept:

- Keep or restore the wizard in the original/start room 0x88.
- When the player is close enough to the wizard, show HTML-side dialogue near
  or below the emulator.
- The dialogue is not drawn inside the Spectrum display.
- It is an emulator-side overlay/control panel, like our diagnostics, and is
  therefore compatible with the project boundary.

First dialogue role:

- The wizard explains that the curse has spread across the infinite map.
- Each region has its own cauldron.
- The cauldron in a region requests one specific charm.
- The matching charm is somewhere nearby, inside the same sector.

Detection options:

- Conservative first proof:
  - show the dialogue when the current logical room is the starting room.
- Better proof:
  - detect the wizard's active object slot and player proximity in working
    memory.
- Later proof:
  - add a Talk button or key binding that only appears when the player is near
    an NPC.

State:

- Dialogue state lives in JS.
- It should not pause or mutate the game unless explicitly designed to do so.
- It can be used for tutorial, hints, quest status, and debugging.

Risks:

- Wizard object layout and animation slots need careful confirmation.
- Proximity detection may be unreliable if based on object slots before we
  identify the active record consistently.
- A dialogue panel must not hide critical emulator/debug controls.

Exit criteria:

- The player can see a wizard explanation outside the emulator display.
- The original game continues running normally.
- No Spectrum rendering code is modified.


Stage 8.4 - C3 Original Charm/Item/Cauldron Mechanic Probes
-----------------------------------------------------------

Goal:

- Learn how much of the original charm/item/cauldron system can be safely
  reused for dispersed sector quests.
- Preserve the working C1/C2 behaviour:
  - original cauldron visual background is room-local;
  - cauldron bubbles are room-local under `?stage8cbubbles=cauldron`;
  - ordinary rooms remain bubble-free.
- Avoid turning on original global quest completion logic until its state
  bytes are understood.

Known anchors:

- live item/charm slots around 0x5C48 and 0x5C68;
- dynamic records around 0x5C88 + n * 0x20;
- current room id at 0x5C10;
- room working fields around 0x5BAB..0x5BAE.
- original static item/object table around 0x6FF2;
- cauldron background id 0x13 and cauldron visual records 0x8D/0x8E;
- cauldron bubble template at 0xB8C8..0xB8D9;
- live bubble/item-display area around 0x5C68;
- charm codes 0x60..0x66;
- object code 0x67 is extra life/consumable, not a quest charm.

C3.0 - Read-Only Original Mechanic Anatomy
------------------------------------------

Goal:

- Watch the original item, inventory, cauldron, and display state without
  changing gameplay bytes.

Tasks:

- Observe a normal original charm/item pickup if possible.
- Record which working-memory slots change.
- Sample live slots around:
  - 0x5C48 first item/charm slot;
  - 0x5C68 second item/charm/bubble/display slot;
  - 0x5C88 + n * 0x20 dynamic object records.
- Sample static object records around 0x6FF2 before and after:
  - entering a room with an item;
  - picking up an item;
  - carrying an item;
  - leaving a room after moving/dropping an item.
- Determine whether charm identity, carried state, and current room are stored
  in simple per-object fields or coupled to deeper game-state tables.
- Identify candidate bytes for:
  - currently carried item/charm;
  - requested cauldron charm;
  - item pickup/drop state;
  - item writeback to the static table;
  - cauldron item-display state.
- Add diagnostics for:
  - raw item slot bytes;
  - decoded candidate item id;
  - current logical room role;
  - current quest required charm;
  - static item-table readback for nearby records.

Animation/display timing side quest:

- Record the counters or state bytes that drive:
  - bubble animation speed;
  - requested-charm flashing speed.
- Do not slow the whole emulator as a first fix.
- Prefer identifying original animation/display counters first; only then
  decide whether to patch them, mirror them externally, or leave them alone.

Exit criteria:

- We know which bytes change during item pickup/carry/drop.
- We have candidate addresses for the requested-charm display and cauldron
  item-display state.
- We know whether `frameCompleted` is late enough to miss critical item-state
  transitions.
- No item/charm/cauldron gameplay bytes have been deliberately modified yet.

C3.1 - Single Charm-Room Item Injection
---------------------------------------

Goal:

- Place one original item/charm in one generated logical charm room without
  corrupting the sliding map.

Tasks:

- Keep this probe opt-in, currently `?stage84c31=1` or `?stage8c31=1`.
- Test whether a charm can be placed in an injected logical charm room by
  rewriting one controlled original static object-table record before
  `find_special_objs_here` / room loading builds the live item slots.
- Use static object record 31 at 0x7109..0x7111 as a disposable probe record.
- Use one original item/charm id from the pickup-safe 0x60..0x66 range,
  derived from the sector's required charm id.
- Arm the record when a logical charm anchor is present in the current
  five-room physical cross, preferably the center room if it is the charm
  room.
- Set the record inactive with current/start room 0xFF when no logical charm
  anchor is present in the current physical cross.
- Do not directly write inventory bytes, charm-order bytes, completion bytes,
  or JS quest-completion state during this probe.
- Skip static-record reseeding while the carried/inventory queue points back
  to the disposable record, so a carried probe item is not clobbered by
  recentering.
- Keep JS persistent quest state as the source of truth until original carry
  state is understood.
- Test whether leaving the room writes object state back into original static
  tables, and whether we need to intercept or neutralize that writeback.
- For this disposable proof, warn the tester to pick the item back up before
  leaving if it is dropped in a non-cauldron room.

Exit criteria:

- The item appears only in logical charm rooms armed by the current five-room
  cross.
- The item does not appear in unrelated ordinary or cauldron rooms after
  sliding/recentering.
- Leaving/re-entering the room is stable.
- Live-slot pointers confirm whether the object points back to 0x7109.
- Static-table writeback is understood or safely neutralized enough to move to
  C3.2.

C3.2 - Carried/Inventory State Probe
------------------------------------

Goal:

- Determine whether original carried-item state can be reused or whether the
  dispersed quest needs a JS-side shadow inventory.

Tasks:

- Observe what changes when the player picks up the injected item.
- Observe what changes when the player carries it across at least one room
  transition.
- Decode the original carried queue at 0x5BD8..0x5BE7 as four 4-byte slots:
  - displayed sprite;
  - flags/state byte;
  - little-endian pointer back to the static object table.
- Track whether a C3.1 disposable charm keeps pointer 0x7109 -> static object
  31 after:
  - pickup;
  - one or more Stage 7 recenter transitions;
  - arriving at a logical cauldron room.
- Report whether record 31 being zeroed after pickup is harmless because the
  carried display slot still preserves the sprite and static-table pointer.
- Compare original memory state with JS quest metadata:
  - required charm type;
  - local charm anchor;
  - collected charm type;
  - sector completion state.
- Separate two identity levels:
  - exact JS source charm, known when the pointer is the C3.1 disposable
    record and the last seeded logical charm target is still known;
  - original sprite class 0x60..0x66, which is the actual seven-type charm
    identity used by Knight Lore. The 14 original deliveries are two requests
    for each of these seven types, not 14 distinct charm identities.
- Decide whether a charm collected in one sector can be accepted by another
  sector's cauldron using original state, JS state, or a hybrid.
- Do not patch cauldron requested-object display or acceptance in this substage.

Exit criteria:

- We know whether original carry/inventory state is stable across the sliding
  physical-room remap.
- We know whether carried charm identity is explicit enough to support the
  global charm-type policy.
- We know C3.3 can trust carried sprite 0x60..0x66 as the charm type, while
  still using JS metadata for sector ownership, completion state, and source
  bookkeeping.

C3.3 - Cauldron Acceptance Probe
--------------------------------

Goal:

- Test one controlled charm delivery to one generated quest cauldron.

Tasks:

- Use a single sector, ideally sector (0,0), for the first proof.
- Require one known charm type.
- Let the player carry or JS-mark that charm as collected.
- Accept only carried sprites 0x60..0x66 as charm types; reject 0x67 because
  it is extra life/consumable.
- Treat the original 14-object objective as two requests for each of the seven
  charm types, not as 14 separate charm identities.
- Enter the sector cauldron room with:
  - original cauldron visual enabled;
  - room-gated bubbles enabled only if desired;
  - diagnostics watching cauldron/request/inventory bytes.
- Probe whether the original cauldron logic can accept the charm without
  corrupting global order/state.
- If original acceptance is too global or fragile, fall back to JS-side
  completion detection while leaving the cauldron visual original.
- First implementation:
  - C3.3 is opt-in with `?stage84c33=1` or `?stage8c33=1`.
  - The runtime accepts a carried charm when the player is in a logical
    cauldron room and the carried sprite is the required 0x60..0x66 charm
    type, but only after a player drop action is detected.
  - A matching carried charm at the cauldron is a `ready` state, not automatic
    completion.
  - Drop detection uses original cauldron/drop state when available
    (`obj_dropping_into_cauldron` or `objects_put_in_cauldron`) and otherwise
    falls back to a rising edge on `pickup_drop_pressed` while `cant_drop` is
    zero.
  - Diagnostics count only ready-state drop edges, so earlier pickup/drop
    edges do not look like cauldron deliveries.
  - Acceptance marks only the external persistent quest-sector state as
    `completed`.
  - With C3.3 alone, the original carried object is not consumed by the JS
    completion marker. If C3.4 is also enabled and original acceptance succeeds,
    the original engine may consume it.
  - Original inventory bytes, charm-order bytes, original completion counters,
    and global quest-order state remain untouched.
  - The original cauldron requested-object display is still independent and
    may disagree with the JS sector request until a later display/order patch.

Exit criteria:

- One sector quest can transition from `charmCollected` to `completed` by
  returning to the cauldron.
- We know whether this transition was driven by original mechanics, JS-side
  quest state, or a hybrid.
- No multiple-cauldron global-state corruption is observed.

C3.4 - Original Cauldron Request Display/Order Patch
----------------------------------------------------

Goal:

- Make the original cauldron's visible requested object and original acceptance
  comparison agree with the current JS sector request.

Tasks:

- Keep this probe opt-in, currently `?stage84c34=1` or `?stage8c34=1`.
- Use the disassembly-proven original request path:
  - `objects_put_in_cauldron` at `0x5BBB` is the current request index;
  - `objects_required` at `0xC27D..0xC28A` is the 14-byte request-order table;
  - `ret_next_obj_required` returns `objects_required[objects_put_in_cauldron]`.
- Patch only the active byte when:
  - the current logical room is a quest cauldron; or
  - the current five-room cross contains a directly reachable adjacent quest
    cauldron, so the request can be pre-armed before the original room-load
    code initializes the visible cauldron display.
- Active byte:
  - address `0xC27D + objects_put_in_cauldron`;
  - value `requiredCharm.id` / sprite index `0..6`.
- If the target JS quest sector is already completed, leave the original
  request slot unchanged so one local cauldron cannot repeatedly advance the
  original global counter during this probe.
- Report:
  - original counter value;
  - active request slot address;
  - previous and desired request index;
  - full 14-byte request table readback;
  - write count and last action.
- Be honest that this is no longer a pure display patch: because the original
  acceptance routine uses the same `ret_next_obj_required` path, a correct drop
  may increment original `objects_put_in_cauldron` and trigger original colour
  cycling.
- Do not yet virtualize or reset the original `objects_put_in_cauldron` counter
  unless testing shows global drift or final-animation risk.

Exit criteria:

- The cauldron reveal/display asks for the same charm type as the JS sector.
- Dropping the matching charm can be observed with both:
  - JS sector completion;
  - original request-order state readback.
- We know whether original colour/bubble/acceptance feedback is useful enough
  to keep, or whether the original counter must be isolated in a later probe.

  - be reset/isolated after each sector completion; or
  - be bypassed for progression while retaining selected original feedback.

C3.5 - Animation, Display Timing, And Respawn Follow-Ups
--------------------------------------------------------

Goal:

- Track the two non-blocking issues revealed by C1/C2 before they become
  confusing during playable quest tests.

C3.5a - Read-Only Timing Probe
------------------------------

Goal:

- Measure the actual animation/display cadence before changing it.

Tasks:

- Do not patch animation speed, display speed, counters, live slots, or emulator
  timing in C3.5a.
- Track the live cauldron bubble/display slot around 0x5C68..0x5C87:
  - current sprite;
  - whether the sprite is a bubble frame 0xA0..0xA7;
  - whether the sprite is a requested-object flash 0xA8..0xAE;
  - frame/sample interval between sprite changes;
  - frame/sample interval between requested-object flashes.
- Compute the expected requested-object display sprite from:
  - JS sector required charm;
  - original request index mapping 0..6;
  - displayed sprite 0xA8 + index.
- Keep a short recent sprite trail so manual testing can see whether the slot
  is cycling every frame, every few frames, or more slowly.
- Watch dynamic records for matching 0xA0..0xAE sprites as supporting evidence,
  but treat 0x5C68 as the primary candidate because C2 showed this slot holds
  cauldron bubble/display state.
- Confirm whether the whole game speed feels normal while only this visual
  cycle appears fast.

Exit criteria:

- Diagnostics can say how many frames a bubble/display sprite phase lasts.
- Diagnostics can say how often a requested-charm flash appears.
- We know whether the fast bubble cycle and fast requested-charm flash are the
  same live-slot behaviour or two separate behaviours.

C3.5b - Safe Slowdown Experiment
--------------------------------

Goal:

- Slow only the confirmed requested-object display cadence, if possible, while
  leaving active bubble hazard logic original.

Tasks:

- Keep this opt-in with a separate URL flag.
- Do not change emulator speed globally.
- First implementation flag:
  - `?stage84c35b=1` enables the experiment;
  - `?stage84c35bhold=6` chooses the target visible hold length, clamped to
    1..30 frames;
  - `?stage84c35bbubbles=human` additionally enables human-form 0xA0..0xA3
    bubble-frame holding.
- Prefer a minimal runtime patch around the confirmed animation/display state:
  - freeze or stretch only harmless display phase changes first;
  - avoid changing collision, wolf-kill, item drop, or cauldron acceptance
    logic until proven separate.
- Failed first implementation:
  - predecessor replay on 0xA0..0xA7 bubble phases did not preserve original
    wolf-kill behaviour;
  - C3.5a showed constant writes, no releases, and a forced/reversed sprite
    trail;
  - therefore direct bubble-phase stretching is rejected for now.
- Safer revision:
  - default mode writes only 0xA8..0xAE requested-object display sprites;
  - optional human-bubble mode also writes 0xA0..0xA3, but only while the
    player is human;
  - write the same observed sprite, not the predecessor sprite;
  - never write 0xA4..0xA7 wolf-attack bubble frames;
  - pause all writes when the player is wolf, transforming, or unknown form.
- Cyclic hold correction:
  - rows such as "visible 231/6" show the holder released once but failed to
    start a fresh cycle;
  - after each release, reset the target so the next eligible sample can begin
    another hold cycle.
- Motion-hold revision:
  - sprite-only holding did not perceptibly slow the visible bubble motion;
  - human-bubble mode now holds the first sampled +0 sprite, +3 Z, and +11 dZ
    values during each hold cycle;
  - request flashes remain sprite-only.
- Compare before/after C3.5a timing rows.
- Preserve room-gated bubble behaviour: bubbles must remain confined to logical
  cauldron rooms when `?stage8cbubbles=cauldron` is used.
- Retest wolf-form bubble danger after every C3.5b change. The same 0x5C68
  live slot is used for both requested-object display and active cauldron
  hazard state.

Exit criteria:

- The requested-object flash is readable without breaking original cauldron
  acceptance.
- Bubble movement/animation is either deliberately left original or slowed
  only in human form without affecting wolf-form hazard behaviour.
- Wolf-form bubble danger is preserved.

C3.5c - Routine-Level Bubble Throttle Probe
-------------------------------------------

Goal:

- Test whether throttling the original bubble updater before drawing is a
  better timing lever than post-frame live-slot pokes.

Tasks:

- Keep this opt-in:
  - `?stage84c35c=1` enables the probe;
  - `?stage84c35cperiod=4` runs one original update frame for every four
    frames by default, clamped to 2..12.
- Suspend C3.5b while C3.5c is enabled so live-slot holding and routine
  patching cannot interfere.
- Patch only the confirmed routine body around `upd_160_to_163`:
  - `0xB8E8`: `CALL dec_dZ_and_update_XYZ`;
  - `0xB8EB`: `CALL next_graphic_no_mod_4`.
- Also patch the confirmed request-display reset in `upd_168_to_175`:
  - `0xB926`: `LD (IX+0),0xA0`, expected bytes `DD 36 00 A0`.
- Read and store the original CALL/reset bytes before patching.
- The first routine-level attempt patched both calls to `NOP NOP NOP`; manual
  testing showed irregular speeds, likely because the remaining routine body
  still ran.
- Early-exit revision:
  - on throttled human-form frames, patch `0xB8E8` to `JP 0xB916`;
  - `0xB916` is the routine's normal `set_wipe_and_draw_flags` exit;
  - this skips motion, graphic advance, dZ reset, and request-display logic as
    one unit;
  - keep `0xB8EB` restored/original.
- Request-hold revision:
  - when the observed live sprite is 0xA8..0xAE, patch `0xB926` to
    `NOP NOP NOP NOP` on throttled frames;
  - restore `0xB926` on active frames so the display returns to 0xA0 normally;
  - this should keep the requested charm visible for the same cadence as each
    bubble phase.
- On active frames, outside logical cauldron rooms, and whenever the player is
  not human, restore the original CALL/reset bytes.
- Show current/original call bytes in diagnostics.

Manual status, 2026 07 21:

- The first routine-level NOP attempt confirmed the target routine, but gave
  irregular human-form bubble speeds.
- The early-exit revision, `0xB8E8 -> JP 0xB916`, produced a stable, readable
  human-form bubble cadence.
- The request-hold revision, `0xB926 -> NOP NOP NOP NOP` on throttled
  `0xA8..0xAE` request-display frames, made the requested charm readable.
- Wolf-form bubbles still kill the player, and the diagnostics show the
  original bytes restored in wolf form.
- Default period 4 is playable but a little slow. Period 3 is the next tuning
  candidate for longer two- or three-sector play tests.

Exit criteria:

- Human-form bubble motion is visibly slower.
- The routine bytes are restored outside cauldron rooms and in wolf form.
- Wolf-form bubbles still kill the player.
- Correct human-form charm drop still works.

C3.5d - Display/Feedback Policy

Stage 8.5 - First Playable Dispersed Quest

Deterministic Placement Rules
-----------------------------

The first version should prefer simple, inspectable rules:

- Use the existing 8x8 chunk as the quest sector.
- Pick cauldron and charm coordinates from deterministic hash streams.
- Reject candidate anchors if:
  - both anchors are the same room;
  - the room would be outside the sector;
  - the room cannot be connected to the traversable world without excessive
    carving;
  - the room is the global starting room 0x88 and the placement would block
    the spawn/center area.
- Prefer placing cauldrons near the sector hub.
- Prefer placing charms away from the cauldron but still in the same connected
  component of the traversable world.
- If deterministic candidate selection fails after N attempts, fall back to:
  - cauldron at the hub;
  - charm at the farthest reachable cell inside the sector, measured through
    the traversable world graph rather than only by in-sector paths.

Charm identity:

- First simple rule:
  - `requiredCharm = hash(sectorX, sectorY, worldSeed) % 7`.
  - Values 0..6 map to original charm sprites 0x60..0x66.
  - 0x67 is extra life/consumable and must not be generated as a required
    quest charm.
- Later:
  - weight charm types by region style;
  - avoid repeating the same charm too often in neighbouring sectors;
  - reserve some charm-type patterns or request order rules for authored
    special sectors.

Memory And Timing Policy
------------------------

Safe now:

- compute quest metadata outside the emulator;
- show diagnostics outside the emulator;
- alter generated room labels/styles/blocks;
- force generated anchor rooms to exist;
- use current Stage 7 static location injection.

Potentially safe after probes:

- place original static cauldron/wizard/charm-like objects in injected rooms;
- virtualize charm object slots;
- detect pickup/drop/carry state in live working memory;
- patch the original cauldron state machine.

High risk:

- duplicating original cauldron logic without understanding global state;
- changing original charm order variables blindly;
- allowing update_special_objs to write virtual object state into original
  static tables unexpectedly;
- injecting invalid block/background records;
- changing active room data while retrieve_screen is decoding it.

Timing rule:

- All static location entry rewrites must happen before retrieve_screen reads
  the room.
- Quest metadata can be updated at frameCompleted.
- Active working-memory item/object pokes may need a tighter hook than
  frameCompleted if they race the original game logic.


Open Questions
--------------

- Can multiple cauldrons be represented as original in-game objects without
  confusing the original quest machinery?
- Is the cauldron object purely visual until the player interacts with it, or
  does it participate in special logic as soon as the room loads?
- Are charms keyed by object record, graphic id, item id, or separate state?
- Can the original inventory/charm order be overridden cleanly, or should the
  first dispersed quest implementation remain JS-side?
- Should every 8x8 sector contain a quest, or should quests be sparse so that
  exploration breathes?
- Should the start room wizard be restored only in the origin sector, or should
  wizard/NPC rooms appear in other authored sectors later?


Recommended Implementation Order
--------------------------------

1. Add quest-sector metadata generation.
2. Add diagnostics only.
3. Force cauldron/charm anchor rooms to exist.
4. Add reachability checks and fallback carving.
5. Add JS-side quest state.
6. Add starting-room wizard dialogue panel.
7. Implement first abstract quest loop:
   - visit charm room;
   - return to cauldron room;
   - mark sector complete.
8. Probe original charm/item/cauldron memory.
9. Replace abstract markers with real original objects only where proven safe.


Success Definition
------------------

The stage is successful when:

- the infinite map remains stable under Stage 7 sliding;
- each generated 8x8 sector can define a deterministic local quest;
- the cauldron and charm rooms are reachable;
- quest progress persists outside the emulator;
- the player can understand the new mechanic through external wizard dialogue;
- no original source code or renderer is modified.

The long-term ideal is a hybrid:

- the original Knight Lore engine still renders and simulates the active room;
- JavaScript supplies the infinite world, local quests, persistence, and
  dialogue;
- the player experiences a new dispersed adventure built on top of the real
  1984 game.
