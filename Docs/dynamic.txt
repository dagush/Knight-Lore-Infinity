Knight Lore dynamic working-memory reference
============================================

Source analysed:
- Memoria_RicardGalvany_short.pdf, chapter 4 only.
- Chapter 4 is pages 1..24 of the shortened PDF.
- Chapter 5 starts on page 25 and was not read for this pass.

Purpose
-------
This file condenses the chapter-4 information about Knight Lore's working
memory: how the game expands static room definitions into the dynamic scratch
area, which bytes are useful for visual reconstruction, and which discovered
player fields live in the same memory block.

Scope warning
-------------
Chapter 4 explains the dynamic room build process and several useful addresses,
but it does not fully document every game-internal byte. Preserve raw bytes for
anything not explicitly defined here.

Known dynamic memory range
--------------------------
- 0x5BA0..0x6107: scratch/work memory.
- This memory changes during play and reflects the current room state.
- Chapter 4 identifies this range as the place to inspect for dynamic location
  changes.
- The dynamic room/object data built by RetrieveScreen starts at 0x5C88.

Relationship to static data
---------------------------
Static room data gives only the initial state of a room. It is compact:
- room id.
- stride/displacement to the next room entry.
- dimensions and colour/attribute.
- background identifiers.
- object identifiers plus repeated low-resolution positions.

That is not sufficient for rendering the room after gameplay begins. Objects can
move, be collected, disappear, or change state. The game therefore expands the
current room into work memory and updates that expanded copy while the player is
inside the room.

RetrieveScreen summary
----------------------
Chapter 4 names the relevant routine as RetrieveScreen.

High-level flow:
1. Start at the first static room entry.
2. Advance through static entries until the current room id is found.
3. Read room attributes.
4. Store the room/location attributes in working memory.
5. For each background:
   - resolve its static background data from the background table at 0x6CE2.
   - read sprite id, dimensions, position, and representation flags.
   - calculate/store the dynamic sprite data.
6. For each object:
   - resolve its static object/block data from the object table at 0x6BD1.
   - read sprite id, dimensions, low-resolution position, flags, and correction
     data.
   - calculate the high-resolution 3D position.
   - store the dynamic sprite data.

Key interpretation:
- Dynamic memory stores expanded sprite instances, not compact object types.
- If a static object type has four room positions, the dynamic memory has four
  separate object instances.
- Backgrounds and objects may each be made of multiple sprites.

Dynamic visual sprite record
----------------------------
Chapter 4 figure 4.19 identifies the useful visual record as 8 bytes:

- byte 0: sprite number.
- byte 1: high-resolution X position.
- byte 2: high-resolution Y position.
- byte 3: high-resolution Z position.
- byte 4: size/dimension X.
- byte 5: size/dimension Y.
- byte 6: size/dimension Z.
- byte 7: representation info/flags.

The record applies to both background sprites and object sprites after
RetrieveScreen has expanded the current room.

Examples from chapter 4:
- East arch first sprite:
  - 0x5C88: 02 C4 73 80 05 03 28 00
- East arch second sprite:
  - 0x5CA8: 03 C4 8D 80 05 03 28 00
- Fire object instances:
  - 0x5EC8: B4 68 79 80 06 06 0C 30
  - 0x5EE8: B4 98 89 80 06 06 0C 30
  - 0x5F08: B4 88 8B 80 06 06 0C 30
  - 0x5F28: B4 78 99 80 06 06 0C 30

Dynamic slot spacing
--------------------
Strict reading of figures 4.14, 4.18, and 4.20:
- The useful visual record is an 8-byte group.
- In the shown full room dump, visual records appear every 0x20 bytes:
  - 0x5C88, 0x5CA8, 0x5CC8, 0x5CE8, ...
- The bytes around those highlighted 8-byte visual records are game-internal
  data. Chapter 4 explicitly says those extra bytes were not needed for the
  student's 3D visualization.

Recommendation:
- For visual reconstruction, preserve the full 0x20-byte slot but initially use
  the 8-byte visual group at addresses 0x5C88 + n * 0x20.
- Do not parse every consecutive 8 bytes from 0x5C88 as a separate visual
  sprite. That would mix visual records with game-internal state.
- Do not use an all-zero 8-byte group alone as a terminator until this has been
  verified against live rooms; the chapter does not define the termination rule.

Dynamic room dump example
-------------------------
Figure 4.20 labels the expanded room data by semantic groups:
- 0x5C88 and 0x5CA8: east arch visual records.
- 0x5CC8 and 0x5CE8: west arch visual records.
- 0x5D08..0x5EA8: wall visual records.
- 0x5EC8..0x5F28: four fire visual records.

The example demonstrates that a compact static room definition becomes a
sequence of expanded per-sprite dynamic records.

3D low to 3D high coordinates
-----------------------------
Chapter 4 states that static coordinates begin in low-resolution 3D, while the
working memory stores high-resolution 3D coordinates used by Filmation.

Conversion:
- X_H = X_L * 16 + X_1 * 8 + 72
- Y_H = Y_L * 16 + Y_1 * 8 + 72
- Z_H = Z_L * 12 + Z_1 * 4 + ScreenZ
- ScreenZ = 128

Meaning:
- X_L, Y_L, Z_L are low-resolution static coordinates.
- X_1, Y_1, Z_1 are correction values stored with the sprite/object definition.
- Some regularly moving objects, such as guards, require these correction values.

Practical consequence:
- Positions inside the dynamic visual record are already high-resolution
  positions.
- Do not reapply the low-to-high conversion to bytes 1..3 of a dynamic visual
  record.

Axis orientation
----------------
Chapter 4 uses experimental edits to static object coordinates to determine the
3D axis orientation. Figure 4.27 shows:
- Z rises vertically on screen.
- X and Y are the two isometric floor directions.

For the current parser, the most important point is that the dynamic records
already contain the high-resolution coordinates used by the game renderer.

Backgrounds in working memory
-----------------------------
For each static background id in the current room:
- RetrieveScreen looks up the background pointer table at 0x6CE2.
- It reads one or more static background sprite records.
- It writes one dynamic visual record per resulting sprite.

Static background data format, repeated here because RetrieveScreen expands it:
- byte 0: sprite id.
- byte 1: X position.
- byte 2: Y position.
- byte 3: Z position.
- byte 4: width/dimension X.
- byte 5: depth/dimension Y.
- byte 6: height/dimension Z.
- byte 7: representation flags.

Open point:
- Chapter 4 says byte 7 includes horizontal/vertical rotation flags, while the
  earlier static data document describes vertical/horizontal flip bits. Treat
  byte 7 as raw representation flags until the bit meanings are verified.

Objects in working memory
-------------------------
For each static object descriptor in the current room:
- RetrieveScreen looks up the object/block pointer table at 0x6BD1.
- It expands each repeated low-resolution object position into an individual
  dynamic instance.
- It writes one dynamic visual record per resulting sprite.

Static object data used during expansion:
- byte 0: sprite id.
- bytes 1..3: object dimensions X, Y, Z.
- byte 4: flags for rotation, movement, and dependency on another object.
- byte 5: position-correction data.

Strict note:
- Chapter 4 prose says object data is made of "groups of 6 bytes", but the fire
  example includes a trailing 0x00 terminator:
  - B5 06 06 0C 10 00 00
- Treat this as a 6-byte record plus sprite-zero terminator, matching the static
  data document.

Fire example
------------
Static room 0x0E example:
- object descriptor: 0x53.
- object id: 0x0A, fire.
- count bits indicate 4 instances.
- low-resolution positions:
  - 0x12 -> x=2, y=2, z=0
  - 0x1D -> x=5, y=3, z=0
  - 0x2C -> x=4, y=5, z=0
  - 0x23 -> x=3, y=4, z=0

Dynamic result:
- The four fire instances become four separate dynamic visual records.
- Chapter 4 highlights these at:
  - 0x5EC8
  - 0x5EE8
  - 0x5F08
  - 0x5F28

Player working-memory fields
----------------------------
Chapter 4 discovers the main character by comparing work-memory captures while
moving the character.

Body position, according to prose:
- 0x5C09: body X.
- 0x5C0A: body Y.
- 0x5C0B: body Z.
- Example value: 80 80 80.

Head position, according to prose:
- 0x5C20: head X.
- 0x5C2A: head Y.
- 0x5C2B: head Z.
- Example value: 80 80 8C.

Strict discrepancy:
- Figure 4.28 visually shows the 80 80 8C triple at 0x5C29..0x5C2B, not
  0x5C20..0x5C2B.
- 0x5C20 contains another byte in the shown dump.

Recommendation:
- Preserve the chapter-4 prose mapping:
  - semantic head X candidate = 0x5C20.
- Also preserve the figure/runtime mapping:
  - render head X candidate = 0x5C29.
- Use 0x5C29 for visual overlay tests until confirmed otherwise.

Player orientation fields
-------------------------
Chapter 4 identifies four bytes relevant to player orientation:
- 0x5C41: one player sprite id byte.
- 0x5C45: another player sprite id byte.
- 0x5C0F: mirror/inversion flag byte.
- 0x5C2F: mirror/inversion flag byte.

The prose says:
- 0x5C41 and 0x5C45 identify the head/body sprite ids.
- The same sprite id can represent multiple directions if the graphic engine
  mirrors it horizontally.
- North uses the same sprite as west, horizontally inverted.
- South uses the same sprite as east, horizontally inverted.
- 0x5C0F and 0x5C2F distinguish mirrored vs non-mirrored presentation.

Strict note:
- Chapter 4 does not define exact bit meanings for 0x5C0F and 0x5C2F.
- Figure 4.30 shows example values 0x7C and 0x7E.
- Do not interpret these as booleans without verification.
- Chapter 4 points to section 5.6 for more detail, but section 5.6 was not read
  in this pass.

Connection to current project notes
-----------------------------------
The current project notes already track:
- body position at 0x5C09, 0x5C0A, 0x5C0B.
- head semantic position at 0x5C20, 0x5C2A, 0x5C2B.
- head render-X candidate at 0x5C29.
- mirror flags at 0x5C0F and 0x5C2F.
- sprite ids at 0x5C41 and 0x5C45.

Chapter 4 supports these anchors, with the explicit caveat that 0x5C20 vs
0x5C29 must remain labelled as a discrepancy until resolved.

Useful decoder outline
----------------------
1. Capture work memory 0x5BA0..0x6107.
2. Read room-level fields from known project anchors, but keep raw bytes visible.
3. For expanded room sprites:
   - start visual scan at 0x5C88.
   - read 8-byte visual groups at 0x5C88 + n * 0x20.
   - fields are sprite, x, y, z, sizeX, sizeY, sizeZ, representationFlags.
   - preserve the full surrounding 0x20-byte slot for later reverse engineering.
4. Stop condition remains unconfirmed by chapter 4.
5. Read player:
   - body: 0x5C09, 0x5C0A, 0x5C0B.
   - head semantic candidate: 0x5C20, 0x5C2A, 0x5C2B.
   - head render candidate: 0x5C29, 0x5C2A, 0x5C2B.
   - sprite ids: 0x5C41, 0x5C45.
   - mirror flags: 0x5C0F, 0x5C2F.
6. For visual rendering, use dynamic record positions as high-resolution
   coordinates directly.

Open verification points
------------------------
- Dynamic slot stride:
  Recommendation: use 0x20 between visual records because figures 4.14, 4.18,
  and 4.20 show that pattern, but verify across several rooms.
- Dynamic termination:
  Recommendation: do not infer termination from one all-zero 8-byte group. Use
  the whole 0x5BA0..0x6107 range or a room-builder-derived count until tested.
- Player head X:
  Recommendation: keep both 0x5C20 and 0x5C29 labelled. Use 0x5C29 for render
  overlay only, not as a replacement for the student's prose mapping.
- Representation flags:
  Recommendation: expose raw byte 7 for dynamic visual records; only name bits
  after checking against chapter 5 or runtime experiments.
