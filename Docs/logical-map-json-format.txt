Knight Lore Infinity Logical Map JSON Format
===========================================

Current format id:

    knight-lore-infinity-logical-map-v1

Purpose
-------

This JSON format describes the external logical map used by the JavaScript
side of Knight Lore Infinity. It is not original game source code and it is
not a renderer format. It is an editable authoring format that can compile a
logical room into the compact Knight Lore location-entry bytes used by the
original engine.

The same format is used by:

- user-authored maps;
- generated/deterministic logical rooms;
- the original Knight Lore room dump in:

      SRC/static/maps/knight-lore-original-map.json

Local test URL:

    http://127.0.0.1:8000/index.html?map=maps/knight-lore-original-map.json


Coordinate Convention
---------------------

Logical coordinates are integer pairs:

    { "x": 0, "y": 0 }

For the original-room dump, room 0x88 is the logical origin:

    0x88 -> (0, 0)
    0x89 -> (1, 0)
    0x87 -> (-1, 0)
    0x98 -> (0, 1)
    0x78 -> (0, -1)

North is y + 1. South is y - 1.

The original physical room id is not the logical coordinate. It can be kept as
a label or as originalRoomId for provenance.


Top-Level Document
------------------

Preferred document shape:

    {
      "format": "knight-lore-infinity-logical-map-v1",
      "title": "My custom map",
      "coordinateConvention": "north is y + 1; south is y - 1",
      "origin": { "label": "start", "coord": { "x": 0, "y": 0 } },
      "notes": [
        "Optional human notes."
      ],
      "rooms": [
        ...
      ]
    }

The runtime also accepts a bare JSON array of rooms, but the object form above
is preferred because it can carry title, notes, and provenance.

Top-level fields:

- format:
  Format identifier. Use "knight-lore-infinity-logical-map-v1".

- title:
  Human-readable map title.

- coordinateConvention:
  Human-readable convention note. The runtime currently assumes north is y + 1.

- origin:
  Optional human/provenance field naming the intended starting room.

- notes:
  Optional string array for editor notes.

- rooms:
  Array of room definitions.


Infinite and Finite Maps
------------------------

The logical map model is coordinate-based, not array-size-based. The runtime
can ask for any integer coordinate:

    logicalMap.getRoomAt(12, -4)

If an authored room exists at that coordinate, the authored room is used. If
no authored room exists, the current Stage 7 provider creates a deterministic
procedural answer from the coordinate. The answer may be an open generated
room or a missing/closed cell. That means the generated map is algorithmically
unbounded: revisiting the same coordinate gives the same result, and only
browser memory, save-file size, and practical integer ranges limit how much
persistent state can be kept.

Current generated provider:

    SRC/runtime/knightlore-mapgen.js

Current algorithm:

- split the plane into deterministic 8x8 chunks;
- choose edge connector positions from shared edge seeds so neighbouring
  chunks agree where cross-chunk doors can exist;
- carve guaranteed paths through each chunk to keep the infinite map
  traversable;
- add deterministic corridor branches from the connected core until the local
  room density is close to the original game's 128/256 occupied cells;
- use original-game rectangular room shapes only for legal small-side exits:
  north/south rooms use selector 1, east/west rooms use selector 2;
- throttle legal rectangular candidates deterministically so the generated
  rectangle ratio stays close to the original game's 58/128 rooms;
- assign deterministic style/biome regions that influence generated room
  colour, stone/tree theme, simple block dressing, labels, and metadata;
- keep corner, T-junction, and four-exit generated rooms square;
- return missing cells as closed wall-only payloads, not as open rooms.

Exits are then derived from room existence:

    east exit exists if room(x + 1, y) exists

and similarly for the other cardinal directions. This keeps generated exits
reciprocal by construction before the Stage 7 reciprocal safety overlay runs.

A finite map uses the same structure. It is not a different file format.
Authors make a finite edge by omitting exits at the boundary, or by providing
a map/provider policy that closes exits where no room should be reachable.
The sliding physical five-room cross is unchanged; only the logical room
provider decides whether a neighbour exists as an open room or a closed wall.


Room Definition
---------------

Minimal high-level room:

    {
      "label": "start",
      "aliases": ["0x88"],
      "title": "Starting room",
      "coord": { "x": 0, "y": 0 },
      "size": { "selector": 0 },
      "colour": 6,
      "theme": "stone",
      "exits": {
        "north": "arch",
        "east": "arch",
        "south": "arch",
        "west": "arch"
      },
      "blocks": [
        {
          "type": 3,
          "positions": [
            { "x": 2, "y": 3, "z": 0 },
            { "x": 5, "y": 3, "z": 0 }
          ]
        }
      ],
      "objects": [],
      "items": []
    }

Room fields:

- label:
  Required in practice. A unique human-readable room name. Examples:
  "start", "0x88", "cauldron", "west-gallery".

- aliases:
  Optional array of additional unique names. The runtime can look rooms up by
  label or alias. In the original dump, room 0x88 also has alias "cauldron".

- title:
  Optional display name. If omitted, the label is used.

- coord:
  Required integer logical coordinate.

- originalRoomId:
  Optional provenance field for rooms copied from the original game. It is not
  used as the logical coordinate.

- size:
  Required object with selector.

- colour:
  Original low 3-bit room attribute/colour value, 0..7.

- theme:
  Optional high-level background generator hint. Currently "stone" or "tree".
  Theme is only used when backgrounds is omitted.

- exits:
  Semantic exit map. Use false for no exit or "arch" for an arch exit. Future
  transition logic may use this field as map connectivity.

- backgrounds:
  Optional exact compact background id list. If present, it is authoritative
  for compilation and theme is ignored. This is how the original-room dump
  preserves exact original wall, arch, portcullis, wizard, cauldron, and high
  arch background ids.

- blocks:
  Compact foreground block runs, represented as block type plus grid
  positions. If blocks is empty, the compiler must not emit a 0xFF separator
  after the backgrounds; see "Compilation to a Physical Location Entry" below.

- objects:
  Optional dynamic object authoring/state placeholder. Accepted by the runtime
  and persisted outside the emulator, but not compiled into location-entry
  bytes yet.

- items:
  Optional item authoring/state placeholder. Accepted by the runtime and
  persisted outside the emulator, but not compiled into item tables yet.

- meta:
  Optional provenance/debug object. The compiler ignores it. It is safe to
  delete meta when copying a room into a custom map.


Labels and Uniqueness
---------------------

The loader rejects:

- duplicate logical coordinates;
- duplicate labels;
- duplicate aliases;
- a label or alias that points to two different coordinates.

Examples:

    window.KnightLoreInfinity.logicalMap.getRoomByLabel("0x88")
    window.KnightLoreInfinity.logicalMap.getRoomByLabel("cauldron")


Room Sizes
----------

Supported size selectors:

    selector 0: 64, 64, 128  square room
    selector 1: 32, 64, 128  rectangular room
    selector 2: 64, 32, 128  rectangular room

Door rule for rectangular rooms:

- selector 1 uses north/south exits and wall id 0x0E;
- selector 2 uses east/west exits and wall id 0x0D;
- avoid long-side doors in rectangular rooms.

The current procedural generator only creates rectangular rooms when the exit
mask uses the narrow side(s) of a rectangle:

    north, south, or north + south -> selector 1 candidate
    east, west, or east + west     -> selector 2 candidate

Legal candidates are not always rectangular. They are throttled by a
deterministic coordinate hash so that the generated rectangle ratio is close
to the original data:

    original map occupancy:   128 / 256 cells = 50%
    original rectangle ratio: 58 / 128 rooms  = 45.31%

If a generated room cannot be represented with small-side doors only, or if
the deterministic throttle leaves it square, it falls back to a square room
with the corresponding doors.

Generated rectangular rooms have no procedural block runs. Stone rectangular
rooms use the original rectangular stone arch/wall combinations.
Tree-themed rectangular rooms are experimental. During Stage 7, the runtime
installs custom tree-wall background definitions into the unused tail of the
erased location table and repoints ids 0x12/0x13 to them:

    0x12 custom tree wall size 2 -> selector 2
    0x13 custom tree wall size 3 -> selector 1

Tree rectangular rooms then use tree arch ids 0x04..0x07 on legal exits plus
the custom tree wall id for their selector. This is Knight Lore-compatible
injected data, but it is not a rectangular wooden pattern found in the
original room dump. The custom tree-wall data keeps only source records that
already use Z/base byte 0x80. Raised stone-wall layers are discarded rather
than moved down, because lowering them can cover otherwise valid arches.

To fill the doorless long side, rectangular tree rooms now copy the complete
8-record square-room wall pattern on the relevant axis, then translate only
the room-size plane:

    selector 1 -> full west wall pattern at x = 0x5F
    selector 2 -> full north wall pattern at y = 0xA0

The runtime removes the matching stone-derived ground wall records from the
custom background before appending the translated tree records. This avoids
duplicate overlapping wall panels.

Selector 1 custom 0x13 west-wall records:

    0x80 0x5F 0x49 0x80 0x00 0x08 0x2C 0x10
    0x81 0x5F 0x58 0x80 0x00 0x08 0x2C 0x10
    0x82 0x5F 0x68 0x80 0x00 0x08 0x2C 0x10
    0x80 0x5F 0x78 0x80 0x00 0x08 0x2C 0x10
    0x81 0x5F 0x88 0x80 0x00 0x08 0x2C 0x10
    0x80 0x5F 0x98 0x80 0x00 0x08 0x2C 0x10
    0x81 0x5F 0xA8 0x80 0x00 0x08 0x2C 0x10
    0x82 0x5F 0xB8 0x80 0x00 0x08 0x2C 0x10

Selector 2 custom 0x12 north-wall records:

    0x80 0x48 0xA0 0x80 0x08 0x00 0x2C 0x50
    0x81 0x58 0xA0 0x80 0x08 0x00 0x2C 0x50
    0x82 0x68 0xA0 0x80 0x08 0x00 0x2C 0x50
    0x80 0x78 0xA0 0x80 0x08 0x00 0x2C 0x50
    0x81 0x88 0xA0 0x80 0x08 0x00 0x2C 0x50
    0x80 0x98 0xA0 0x80 0x08 0x00 0x2C 0x50
    0x81 0xA8 0xA0 0x80 0x08 0x00 0x2C 0x50
    0x82 0xB8 0xA0 0x80 0x08 0x00 0x2C 0x50

Earlier attempts to lower raised stone layers, copy only partial filler rows,
or hand-fill the wall with new explicit records either covered valid arches,
left holes, or caused unstable player/emulator behaviour and were reverted.

A later attempt to hand-fill the long doorless walls with new explicit records
caused unstable player/emulator behaviour and was reverted.

Generated wooden rooms use square-room tree background patterns copied from
original rooms by exit mask. If a generated or reciprocal-overlay exit mask
has no known safe tree pattern, the compiler falls back to a stone room for
that compiled payload rather than guessing an unsafe tree filler/arch
combination.


Starting Room Safety
--------------------

The starting room must have a clear spawn area. If the player starts inside a
blocking foreground block, blocking object, or large central background, the
original engine can leave the character trapped immediately after room load.
This happened with the original room 0x88 before the wizard/cauldron expansion
data was removed.

Authoring rule:

- keep the center of the origin room empty;
- do not place foreground blocks on the central floor cells;
- do not place blocking dynamic objects/items at the spawn point;
- avoid central blocking background ids such as 0x12 wizard and 0x13 cauldron
  in the starting room unless the runtime deliberately disables their expanded
  data.

Until the exact player collision footprint is fully documented, use this
conservative grid clearance for the origin room:

    no block at z=0 on x=3..4, y=3..4

It is fine to place decorative or blocking content elsewhere in the starting
room, including around the center, as long as the player has a clear initial
standing area and can walk to at least one intended exit.


Reciprocal Exit Safety
----------------------

For sliding-window play, a visible door must have a matching return door in
the contiguous logical room. Otherwise the original engine can transition into
a room whose wall/background data does not contain the entrance side, leaving
the player apparently behind a wall.

The Stage 7 runtime compiles a reciprocal-exit overlay before writing the
physical five-room cross. The source JSON is not modified. For each adjacent
pair:

- if either authored room defines a door, both compiled rooms get the matching
  door when both room shapes allow it;
- if a generated room borders an authored room, the authored room controls the
  boundary, so generated four-exit defaults do not punch new holes into an
  authored map edge;
- if either room shape cannot legally host that door direction, the compiled
  pair is closed rather than emitting an invalid rectangular-room door.

The original dump intentionally remains a faithful dump. It contains some
one-way exits in raw JSON, so use the Stage 7 diagnostics row to see whether
the reciprocal overlay added or removed exits in the current physical cross.


Backgrounds
-----------

If backgrounds is omitted, the runtime builds backgrounds from theme + exits:

- stone + selector 0:
  arch ids 0x00..0x03 and wall id 0x0C.

- tree + selector 0:
  original-derived square-room tree patterns by exit mask. Do not assume every
  wooden room uses all filler ids 0x0F, 0x10, and 0x11. For example, a safe
  four-exit wooden room uses:

      0x04 0x05 0x06 0x07 0x0F

  rather than:

      0x04 0x05 0x06 0x07 0x0F 0x10 0x11

- rectangular stone:
  selector 1 wall id 0x0E;
  selector 2 wall id 0x0D.

Known background ids:

    0x00 arch north
    0x01 arch east
    0x02 arch south
    0x03 arch west
    0x04 tree arch north
    0x05 tree arch east
    0x06 tree arch south
    0x07 tree arch west
    0x08 portcullis north
    0x09 portcullis east
    0x0A portcullis south
    0x0B portcullis west
    0x0C wall size 1
    0x0D wall size 2
    0x0E wall size 3
    0x0F tree room size 1
    0x10 tree filler west
    0x11 tree filler north
    0x12 wizard
    0x13 cauldron
    0x14 high arch east
    0x15 high arch south
    0x16 high arch east base
    0x17 high arch south base

Stage 7 custom-background caveat:

- in the original data, 0x12 and 0x13 are wizard and cauldron backgrounds;
- Stage 7 repoints 0x12 and 0x13 at runtime to custom rectangular tree-wall
  definitions after disabling the original wizard/cauldron data;
- authored maps that need the original wizard/cauldron visuals should not rely
  on those ids while the Stage 7 custom wall experiment is active.

Important:

- If backgrounds is present, it is used exactly.
- In that case exits is still useful semantic/navigation data, but it does
  not generate or alter background ids.
- If editing a room copied from the original dump, keep backgrounds and exits
  consistent.


Blocks
------

Blocks are represented as runs:

    {
      "type": 3,
      "positions": [
        { "x": 2, "y": 3, "z": 0 },
        { "x": 5, "y": 3, "z": 0 }
      ]
    }

Rules:

- type must be an integer 0..31.
- x and y must be integers 0..7.
- z must be an integer 0..3.
- Runs with more than 8 positions are split into multiple compact
  block/count groups by the compiler.

Known block type names:

    0x00 block
    0x01 fire unused
    0x02 ball up/down
    0x03 rock
    0x04 gargoyle
    0x05 spike block
    0x06 pushable chest
    0x07 pushable table
    0x08 guard east/west
    0x09 ghost
    0x0A fire north/south
    0x0B block high
    0x0C ball up/down offset xy
    0x0D guard square patrol
    0x0E block moving east/west
    0x0F block moving north/south
    0x10 movable block
    0x11 spike block high
    0x12 spike ball fall
    0x13 spike ball high fall
    0x14 fire east/west
    0x15 dropping block
    0x16 collapsing block
    0x17 ball bouncing
    0x18 ball up/down
    0x19 spell repels player
    0x1A portcullis up/down
    0x1B portcullis up/down
    0x1C ball up/down offset x


Compilation to a Physical Location Entry
----------------------------------------

For rooms with foreground block groups, the compiler turns a logical room into:

    id, entrySize, header, backgrounds..., 0xFF, block/count groups...

For background-only rooms, the compiler omits the 0xFF separator:

    id, entrySize, header, backgrounds...

This matches original Knight Lore data such as room 0x77. The original
retrieve_screen routine can safely finish a background-only entry when the
entry byte counter reaches zero; adding a 0xFF to an otherwise empty foreground
section can make it read the next room id as block data.

Header:

    header = (size.selector << 3) | (colour & 0x07)

Block/count group:

    groupHeader = (block.type << 3) | (positionCount - 1)
    positionByte = (z << 6) | (y << 3) | x

The physical id is supplied at compile time. For the center room proof this is
usually 0x88:

    window.KnightLoreInfinity.logicalMap.compileRoomAt(12, -4, 0x88)


Runtime API
-----------

Useful browser console calls:

    window.KnightLoreInfinity.logicalMap.getRoomAt(12, -4)
    window.KnightLoreInfinity.logicalMap.getRoomByLabel("0x88")
    window.KnightLoreInfinity.logicalMap.getRoomByLabel("cauldron")
    window.KnightLoreInfinity.logicalMap.compileRoomAt(12, -4, 0x88)
    window.KnightLoreInfinity.loadLogicalMapFromUrl("maps/knight-lore-original-map.json")

The loader validates every authored room. If loading fails, the previous map is
restored.


Original Dump Notes
-------------------

The original-room dump is generated by:

    Tools/dump-original-map.mjs

Source:

    Docs/knightlore.asm location_tbl

Output:

    SRC/static/maps/knight-lore-original-map.json

The dump contains 128 rooms. Labels are original room ids, for example "0x88".
The cauldron room has:

    "label": "0x88",
    "aliases": ["cauldron"]

The generated dump intentionally avoids redundant editable block data:

- blocks have type and x/y/z positions only;
- block names are documented here rather than repeated in every room;
- raw original position bytes are not repeated beside x/y/z;
- exact original location-entry bytes are kept only in meta.locationEntryBytes
  for provenance/debugging.

Some original background-only rooms omit the 0xFF background terminator in the
original compact data. The dump records this in meta.foundBackgroundTerminator.
The logical compiler follows this convention: it emits 0xFF only when foreground
block/count data follows.
