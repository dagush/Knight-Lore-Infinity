Project framing

Started the Knight Lore / JSSpeccy sliding-window virtualization project. The
core rule was set early and kept strict: the original Knight Lore game code and
3D renderer are not to be modified, and the project must use the original
Spectrum/JSSpeccy display only. All experiments are JavaScript-side emulator
work: memory inspection, static table rewriting, working-memory rewriting, and
runtime pokes/patches.

The initial idea was a 3x3 physical room stencil centered around room 0x88, but
that was refined after reviewing the game transition model. Knight Lore room
movement is cardinal only, so a 5-room cross is enough for the first proof:

- center: 0x88
- west: 0x87
- east: 0x89
- south: 0x78
- north: 0x98

This keeps the project smaller and avoids unnecessary diagonal/corner room
assumptions. The external JavaScript layer will own arbitrary logical map
coordinates, while the original game will continue to see a small physical
neighbourhood.

Planning work

Read and merged the existing overview in Docs/Plan 2026 06 27.txt with the
newer static/dynamic memory findings. Created Docs/Plan 2026 06 28.txt as the
current unified staged plan, while leaving the older plan untouched.

The plan established these important anchors and boundaries:

- location table starts at 0x6251
- room size table is at 0x6248..0x6250
- current room anchor is 0x5C10
- current room attribute candidate is 0x5BAD
- current room size bytes are 0x5BAB, 0x5BAC, and 0x5BAE
- player/body slot starts at 0x5C08
- dynamic object slots are 0x20 bytes each
- slot +8 is a room id
- slot +12 carries transition/entry state bits

The staged plan became:

- Stage 0: project foundation
- Stage 1: read-only room loading anatomy
- Stage 2: prove the physical 5-room cross
- Stage 3 onward: room injection, static table rewrite, working-memory patching,
  recentering, and generalization

Stage 1 - Read-Only Room Loading Anatomy

Added a read-only emulator memory bridge so browser-side JavaScript can inspect
the running Spectrum memory without modifying game memory.

Implemented:

- worker-side readMemory message using the existing WASM core peek(addr)
- host-side readMemory(addr, length) Promise API in JSSpeccy
- host-side frameCompleted event emission after each completed emulated frame
- public pass-throughs for readMemory and on() from the JSSpeccy wrapper
- cache-busting for jsspeccy.js and the worker so diagnostics do not silently
  use stale bundles

Built a Stage 1 diagnostics table directly below the emulator display. The
table is read-only and samples live memory after frameCompleted. It shows:

- sampler status and frame/sample counters
- current location from 0x5C10
- observed secondary room byte from 0x5C30
- player body XYZ
- player head XYZ
- slot +12 flags for the body/head slots
- 0x5BAB, 0x5BAC, 0x5BAD, and 0x5BAE room work bytes
- decoded location table entry for the visible room
- size selector and attribute byte
- background ids
- decoded block/count groups
- decoded packed block positions
- attribute comparison against 0x5BAD
- size selector comparison against 0x5BAB/0x5BAC/0x5BAE
- latest observed room transition
- frameCompleted timing note

During manual inspection, 0x5C10 proved to be the useful current-location byte.
0x5C30 did not reliably update with room transitions, so it was demoted to an
observed-only diagnostic value. The transition detector and location decoder now
use only 0x5C10 as authoritative.

Updated the diagnostics using the detailed location format documented at:

https://www.icemark.com/dataformats/knightlore/index.html

The table now reflects the 0x6251 locations format:

- byte 0: location/screen id
- byte 1: entry size
- byte 2: size selector and attribute
- following bytes: background ids
- 0xFF terminator
- following bytes: block/count groups

Block/count groups are decoded as:

- type = header byte >> 3
- count = (header byte & 7) + 1
- each following position byte is decoded as zz yyy xxx

This was cross-checked against Docs/knightlore.asm, especially retrieve_screen,
which scans the location table, loads backgrounds, and expands foreground block
groups into working graphical object slots.

Important Stage 1 conclusions

- The JavaScript layer can read live emulator memory from the browser side.
- 0x5C10 is the practical current-room/location anchor.
- 0x5C30 is useful to observe, but should not be treated as current-room truth.
- The location table can be decoded live from memory.
- The location entry's attribute and size selector can be compared against
  working-memory fields.
- The frameCompleted boundary is useful for observation, but it is still a
  post-frame boundary. Future recentering may need an earlier worker/core hook
  if invisible patching after a transition proves too late.

Stage 1 was considered complete after the diagnostics table was manually checked
and judged to show the right room-loading information.

- 0x2F, 0x44, 0xB3, 0x8F

was found once in decompressed snapshot RAM at:

- start_locations: 0xD1E2..0xD1E5

- 0x88, 0x88, 0x88, 0x88

Stage 3 - One-Room Injection

Stage 3 proved that the JavaScript layer can rewrite a compact static room
entry while preserving the original entry size, and that the original Knight
Lore room loader will consume the rewritten data when the room is rebuilt.

A dormant one-room injection test harness was added for room 0x88:

- target entry: 0x67BB..0x67CE
- original entry size: 0x13
- normal mode leaves the current clean 0x88 configuration active
- test mode is enabled by opening:
  http://127.0.0.1:8000/index.html?stage3test=1

The test rewrites only bytes inside the existing 0x88 entry. It does not resize
the location table entry. The test fingerprint changes:

- header: 0x06 -> 0x11
- size selector: 0 -> 2
- attribute: 6 -> 1
- decoded size: 64/64/128 -> 64/32/128
- background/wall style: stone arches/walls -> tree-room/wooden style
- foreground group: 0x00 block: 8 -> 0x03 rock: 8

The diagnostics table confirmed the injected compact entry:

- header: 0x11 sel 2 attr 1
- size: 64/32/128
- counts: bg 7; groups 1; 0x03 rock: 8
- arch summary: tree arch entries

The decisive manual visual result was successful: with ?stage3test=1 and a
fresh room build/new game, the Spectrum display showed a wooden room with the
expected altered exits and eight rocks. This proves the original retrieve_screen
path consumed the JavaScript-rewritten static data and built a real game room
from it.

A visible diagnostics/cache marker was added so stale-page problems are easy to
spot:

- diagnostics build: stage3-url-test-20260629-2
- script cache tag: stage2-20260629-6

Normal mode explicitly reports that the Stage 3 one-room injection test is
disabled and explains that ?stage3test=1 enables it. Test mode explicitly
reports that the room 0x88 entry was rewritten while preserving entry size.

Stage 3 conclusions

- Fixed-size static room-entry rewriting works.
- Attribute/colour and size selector rewrites are visible to the decoder.
- Background/wall/arch ids can be rewritten in the compact entry.
- Simple foreground block runs can be rewritten.
- retrieve_screen will build a visibly different room from the rewritten static
  entry when the rewrite happens before the room is built.
- Changing the compact entry after a room is already expanded does not
  necessarily change the visible room immediately; already-expanded working

Stage 4.5 - Logical Map JSON Authoring

An intermediate Stage 4.5 was added to make the logical map editable outside

Room labels and aliases were added first. A room can now be named with a label

The runtime can look up rooms by label or alias:

A JSON map loader was then added. The preferred JSON format is:

- top-level object with format, title, coordinateConvention, origin, notes,
  and rooms
- room entries with label, aliases, coord, size, colour, exits, backgrounds,
  blocks, objects, items, and optional meta

The runtime can load a JSON map from a URL:

- window.KnightLoreInfinity.loadLogicalMapFromUrl("maps/knight-lore-original-map.json")

The test URL for loading the original dump is:

- http://127.0.0.1:8000/index.html?map=maps/knight-lore-original-map.json

The loader was made deliberately strict:

- duplicate logical coordinates are rejected
- duplicate labels are rejected
- duplicate aliases are rejected
- invalid size selectors, colours, background ids, block types, and block
  positions are rejected
- failed loads roll back to the previously loaded map

This strictness is important because JSON files will be edited by hand.

Today completed Stage 5. The original goal was to prove that the JavaScript
side can control a small physical room set by rewriting runtime/static map

------------------------------------------------
Room-id force experiment

The first Stage 5 experiment was deliberately tiny: write only the live current
room byte at 0x5C10.

The first version rewrote 0x5C10 back to 0x88 immediately after a drawn frame.
The manual result was surprising and useful:

- leaving room 0x88 did not enter the physical neighbour;
- instead, the game generated room 0x88 on the other side.

This showed that 0x5C10 is not merely a label to normalize after a transition.
It is effectively the room id that retrieve_screen uses when deciding which
static location entry to decode.

The experiment was then generalized into a destination dial:

- ?stage5force=0x78
- ?stage5force=0x98
- ?stage5force=0x87
- ?stage5force=0x89
- ?stage5force=0x88

The force-dial logic:

- waits until the game is stably in 0x88;
- reads 0x5C10 after each drawn frame;
- skips writes while 0x5C10 is 0x88;
- skips writes while 0x5C10 is already the forced target;
- otherwise writes only 0x5C10 to the forced target room id.

Manual testing with ?stage5force=0x78 confirmed the model:

- exiting 0x88 in any direction entered/generated 0x78;
- exiting 0x78 north naturally returned to 0x88, because 0x88 is explicitly
  skipped by the force logic;
- exiting 0x78 in other directions forced the destination back to 0x78;
- the player appeared on the correct side of the destination room.

This is strong evidence that transition positioning and room generation can be
controlled separately enough for sliding-window virtualization.

Important 0xFF Background-Only Room Rule

During manual testing, entering room 0x77 reset/crashed the emulator before the
room was shown. This revealed a critical original-data rule.

Original room 0x77 has no foreground block data and its original compact entry
does not contain a 0xFF separator. The first compiler version always emitted
0xFF after the background list. For a background-only room this produced:

- id, size, header, backgrounds..., 0xFF

with no block/count data after the 0xFF. In retrieve_screen, that makes the
engine enter the foreground-object decoder with no foreground bytes left. It
can then read the next room id as block data, which is exactly the kind of
fragile data mistake that can crash or reset the game.

The compiler rule is now:

- if foreground block groups exist:
  id, entrySize, header, backgrounds..., 0xFF, block/count groups...
- if no foreground block groups exist:
  id, entrySize, header, backgrounds...

Room 0x77 now compiles as:

- 77 05 03 00 01 0C

This rule was documented in:

- Docs/info.txt
- Docs/logical-map-json-format.txt

- transition-side positioning still works while forcing destinations;
- the original location table can be erased at runtime;
- a compact authored static table can be written in its place;
- the original engine can scan and decode that injected table;
- background-only entries must follow the original no-0xFF convention.

This validates the core sliding-map assumption: while the player is inside a
room, the original static map can be considered disposable. It only needs to be
valid when the engine leaves the current room and scans for the next location.

Next natural step: use the same injection machinery to rewrite the physical
window whenever the logical coordinate changes.

| 2026 07 03 - Stage 6, Stage 7, and Procedural Map          |

2026 07 03
----------

This entry covers the work after the Stage 5 static-table injection milestone.
The project moved from "we can erase and rewrite the map" to a working
four-direction sliding-window proof with an unbounded deterministic logical
map provider.

The important rule stayed unchanged: no original Knight Lore source code was
modified, and no alternate renderer was added. The original Spectrum/JSSpeccy
display remains the only visual output. All map virtualization is still done
from JavaScript by rewriting the emulator's runtime memory and static location
table.

------------------------------------------------
Stage 6 - One-Direction Recenter Proof

Stage 6 was considered already proven by the earlier 0x5C10 force-destination
experiments.

The key observation was that changing only the current-room byte after a room
has been drawn can control which room retrieve_screen loads on the next
transition. When the player leaves the current room, the original engine scans
the location table for the room id in 0x5C10, decompresses that location entry
into working memory, and then resumes play from the correct transition side.

Manual testing confirmed the expected behaviour:

- leaving physical 0x88 in any direction can force the next generated room;
- entering a forced destination still places the player on the correct side;
- forcing can be skipped for selected rooms, allowing natural transitions back
  to those rooms.

This established that transition positioning, current-room selection, and
static-table content are separable enough for a recentering model.

------------------------------------------------
Stage 7 - Four-Direction Sliding Cross

Implemented the first full sliding physical cross.

The physical room ids remain fixed:

- center: 0x88
- west: 0x87
- east: 0x89
- south: 0x78
- north: 0x98

The logical map is external and coordinate-based. The runtime tracks a logical
center coordinate, initially (0, 0). Whenever 0x5C10 shows that the original
engine has entered one of the four physical neighbour rooms, the JavaScript
side:

1. detects the transition direction from the physical room id;
2. updates the logical center coordinate;
3. compiles the new center plus its four logical neighbours;
4. erases and rewrites the physical five-room location table entries;
5. patches active working-memory room-id bytes from the entered physical
   neighbour back to 0x88.

This makes the original game continue as if the player is always centered
around physical room 0x88, while the external logical coordinate keeps moving.

Manual movement tests covered:

- repeated movement in one direction;
- east/east/west/west style backtracking;
- north/south alternation;
- small loops;
- alternating axes.

The first generated maps exposed a real authoring/runtime problem: a visible
door must have a corresponding return door in the contiguous logical room. If
not, Knight Lore can transition into a room whose wall/background data has no
matching entrance side, leaving the player behind a wall. A reciprocal-exit
overlay was added at compile time. The source logical map is not modified, but
the compiled physical payload is made safe:

- authored exits are mirrored when both room shapes can legally host them;
- generated-to-generated exits are reciprocal by construction;
- generated rooms do not reopen authored map edges on their own;
- illegal rectangular long-side exits are closed rather than emitted.

After this fix, repeated manual walking in all directions worked without the
wall-lock problem.

------------------------------------------------
Logical Map Model and Authored JSON

The logical map model now supports authored rooms, labels, aliases, generated
fallback rooms, and persistent room state outside the emulator.

Important pieces:

- SRC/runtime/knightlore.js owns the logical map facade and Stage 7 physical
  cross compiler.
- SRC/static/maps/knight-lore-original-map.json contains a JSON dump of the
  original Knight Lore room set.
- Docs/logical-map-json-format.txt documents the editable JSON format.

Labels were added so authored rooms can be referred to by names such as
"0x88" or "cauldron". The original map dump gives users a copy/paste source
for building new maps from original rooms, while the runtime still treats room
coordinates as the real addressing system.

The JSON documentation was expanded with:

- finite versus infinite map notes;
- reciprocal-exit safety rules;
- starting-room clearance requirements;
- original-game rectangular room size rules;
- safe wooden-room background combinations;
- the critical background-only no-0xFF compiler rule.

------------------------------------------------
Procedural Infinite Map Provider

Added a dedicated procedural map generator:

- SRC/runtime/knightlore-mapgen.js

The generator answers "what room exists at logical coordinate (x, y)?" without
requiring a finite map array. Its answers are deterministic: the same world
seed and coordinate always produce the same room.

The first version split the plane into deterministic chunks, carved guaranteed
paths between shared edge connectors, and added branch rooms. This proved the
infinite-map idea, but manual play showed the early generator produced too many
dense square/T/four-way areas and too few rectangular rooms.

The generator was then tuned against the original Knight Lore map:

- original occupancy: 128 rooms in a 16x16 map = 50%;
- original rectangular rooms: 58 out of 128 rooms = 45.31%.

The current algorithm:

- uses deterministic 8x8 chunks;
- chooses matching cross-chunk connector positions from shared edge hashes;
- carves paths from the four edge connectors to a deterministic hub;
- forces the origin and its four cardinal neighbours to exist;
- grows deterministic corridor branches until local occupancy is near 50%;
- derives exits from neighbouring-room existence;
- returns missing cells as closed wall-only payloads;
- creates original-style rectangular rooms only when exits are on the smaller
  side(s) of the rectangle;
- uses a deterministic throttle so not every legal rectangle candidate becomes
  rectangular.

The latest measured sample was close to the original proportions:

- generated occupancy: about 49.84%;
- generated rectangular rooms: about 45.12%.

Rectangular generated rooms are intentionally conservative for now:

- selector 1 for north/south-side candidates;
- selector 2 for east/west-side candidates;
- no long-side doors;
- stone-only;
- no procedural block runs.

Square generated rooms may be stone or wooden. Procedural foreground blocks
avoid the central starting floor cells, so a generated origin room does not
trap the player immediately.

A separate explanation was added:

- Docs/map-generation-algorithm.txt

Biome / Style Decisions

The meaning of "biome/style decisions" was clarified. In this project it does
not mean a new renderer, modern graphics, or external scene drawing. It means a
room-dressing policy that still compiles into original Knight Lore compact
location data:

- generated labels and titles;
- procedural metadata;
- room colour/attribute;

- simple block dressing for square rooms;
- rectangular room shape selection where original Knight Lore rules allow it.

A Stage 7 "Current Generated Cross" diagnostics table was added immediately

- physical role and room id;
- logical coordinate;
- generated/authored label;
- biome/style region;

- theme and colour;
- exits;
- block summary;
- compact entry/background summary.

This made it possible to see when a room was logically wooden-wing but still
visually stone because rectangular tree-wall data was not yet available.

------------------------------------------------
Rectangular Room Rules

The generator keeps the original Knight Lore rectangular-room legality rule:

- selector 1 is 32x64x128 and only allows north/south exits;
- selector 2 is 64x32x128 and only allows east/west exits;
- long-side doors are not emitted;
- corners, T-junctions, and four-exit rooms remain square.

Generated rectangular rooms still have no procedural block runs. This keeps the
shape experiment isolated from collision and object persistence issues.

The rectangle density was tuned against the original dump:

- original rectangular rooms: 58 / 128 = 45.31%;
- generated rectangular rooms remain close to this ratio after deterministic
  throttling of legal rectangular candidates.

------------------------------------------------
Wooden Rectangular Room Problem

The original game appears to use the tree/wood backgrounds 0x0F, 0x10, and
0x11 only with square rooms. Rectangular wooden rooms therefore needed custom
runtime background definitions.

Stage 7 already erases and rewrites the original location table. The custom
rectangular tree-wall experiment uses spare space in the unused tail of that
erased table and repoints two background ids at runtime:

- 0x12: custom tree wall size 2, for selector 2 rooms;
- 0x13: custom tree wall size 3, for selector 1 rooms.

This is Stage 7 runtime data only. In the original game data, 0x12 and 0x13
are wizard and cauldron backgrounds. Authored maps that need those original
visuals must not rely on those ids while the Stage 7 custom wall experiment is
active.

The first stable custom tree-wall pass kept only source wall records already
using Z/base byte 0x80. Raised stone-wall layers were discarded rather than
moved down. Moving raised records to ground level could cover valid arches or
create unstable room layouts.

Several unsafe/insufficient attempts were observed and reverted:

- grounding all raised stone wall records covered exits;
- hand-filling long walls with new explicit records caused player spinning and
  emulator hangs;
- copying only a few extra middle records was also unstable;
- using only the two square-room filler records avoided hangs but left visible
  holes in the long doorless walls.

------------------------------------------------
Diagnostics-Driven Wall Tuning

A "Wooden Square Wall Reference" table was added above the Stage 7 cross
diagnostics. It lists the original square wooden wall records from:

- 0x0F tree room size 1;
- 0x10 tree filler west;
- 0x11 tree filler north.

This gave a direct byte-level reference for the safe square-tree wall pattern.

A slider probe was then added for the selector 1 west wall X coordinate. The
initial question was whether the two 0x10 filler-west rows should remain at
the original square-room X 0x3F or move to the translated rectangular wall
plane. Manual testing found that X 0x5F is the correct value for selector 1.

The diagnostics table was then expanded to show the actual live custom wall
records being injected. This made it clear whether a wall segment was missing
from the payload or merely hidden/occluded in the Spectrum view.

| 2026 07 08 - Stage 8 Dispersed Quest Sector Model          |
+============================================================+

2026 07 08

Stage 8 changed the infinite map from a room-generation proof into the first
layer of a new dispersed quest structure.

The core idea is that each 8x8 logical sector can own a local quest:

- a local cauldron anchor room;
- a local charm anchor room;
- one required charm id;
- persistent JavaScript-side quest state.

This is deliberately metadata-only for the first proof. No original charm,
cauldron, wizard, inventory, or cauldron-order memory is modified yet.

------------------------------------------------
Planning Update

A new plan file was added:

- Docs/Plan 2026 07 08.txt

The plan defines "Dispersed Quest Sectors and External Dialogue" as the next
major layer after the Stage 7 sliding cross. It records the idea that Knight
Lore Infinity should not only be a larger map, but a world made from local
quest regions.

The plan was also corrected after discussion:

- the cauldron and charm belong to the same 8x8 sector;
- the route between them does not have to stay inside that sector;
- detours through neighbouring sectors are valid and desirable;
- fairness should mean global traversable-world reachability, not isolated
  sector-local reachability.

This reduces pressure on the generator and makes neighbouring sectors feel more
integrated.

------------------------------------------------
Generator Changes

Stage 8 quest metadata was added to:

- SRC/runtime/knightlore-mapgen.js

The procedural map generator now exposes deterministic sector queries:

- getQuestSectorAt(x, y)
- getQuestRoomInfoAt(x, y)

Each sector currently uses the existing 8x8 chunk grid and contains a quest by
default. The metadata includes:

- sector coordinate;
- sector origin;
- local room coordinate inside the sector;
- style/biome information;
- required charm id from 14 abstract charm labels;
- cauldron anchor coordinate;
- charm anchor coordinate;
- difficulty value;
- placement note.

The first implementation chooses cauldron/charm anchors from already existing
connected cells in the chunk:

- cauldrons are biased toward the chunk hub;
- charms are biased farther from the cauldron;
- if both anchors collide, another existing cell is chosen when available.

Anchor rooms are marked as existing cells, so the metadata never points at a
missing generated room.

Important correction:

- the old central floor-cell block avoidance rule now applies only to the true
  logical start room (0, 0);
- other rooms may use those central cells, though entrance strips still need to
  remain safe in future content-generation work.

  - none;

- meta.quest.

For this first proof, visiting a logical room marks only JavaScript-side state.
